# AGENTS.md — Orchestration Rulebook

This file is the contract every agent working in this repo must follow. It is loaded automatically by Claude Code (via the `CLAUDE.md` symlink) and should be included in context for every subagent spawned by the orchestrator.

`CLAUDE.md` at the repo root is a symlink to this file. Edit only `AGENTS.md`.

---

## 1. Project overview

Korab's personal website — built and maintained by an AI team orchestrated through Claude Code. You interact through `claude` in a terminal; the agents handle everything else.

**Stack:**
- **Astro 6** (TypeScript strict mode) — page routing, SSR via Vercel adapter
- **Tailwind CSS 4** — Vite plugin (`@tailwindcss/vite`), CSS-first config (`@import "tailwindcss"` in `src/styles/global.css`)
- **shadcn/ui** — React islands via `@astrojs/react`; components live in `src/components/`
- **Keystatic** — local-git-backed CMS (`@keystatic/core` + `@keystatic/astro`). Content at `src/content/posts/*/`. Admin UI at `/keystatic`.
- **MDX** — `@astrojs/mdx` for MDX support alongside Keystatic's markdoc format
- **Vercel** — `@astrojs/vercel@10` adapter, `output: 'server'`

**Directory layout:**
```
src/
  pages/         — Astro routes
  components/    — React/Astro components (barrel: src/components/index.ts, auto-generated)
  content/posts/ — Keystatic-managed blog posts (one subdirectory per post)
  layouts/       — BaseLayout.astro
  lib/           — Shared utilities (cn helper, etc.)
  styles/        — global.css (Tailwind entry point)
keystatic.config.ts   — Keystatic collection/schema definitions
astro.config.mjs      — Framework + integration config
```

**Key decisions locked in:**
- Keystatic collection path uses trailing slash (`'src/content/posts/*/'`) for index-mode storage — required for `index.mdoc` per subdirectory to resolve correctly.
- `@keystatic/astro@5` declared peer dep `astro@"2||3||4||5"` but works fine with Astro 6 in practice.
- TypeScript pinned to `^5` — ecosystem (Astro, Vite) not yet ready for TS6.

## 2. File Ownership Rules

**This is the single most important guardrail in the repo. Read it before spawning any subagent.**

The files and paths listed below are **ORCHESTRATOR-ONLY**. Parallel subagents MUST NOT create, edit, move, or delete them. Changes to these files flow through the orchestrator exclusively, so we never hit an unresolvable merge conflict between two agents racing on the same config.

ORCHESTRATOR-ONLY paths:

- `astro.config.*` and `next.config.*` (any extension — `.mjs`, `.ts`, `.js`, `.cjs`)
- `package.json`
- `pnpm-lock.yaml`
- `tailwind.config.*` (any extension)
- `postcss.config.*` (any extension)
- `tsconfig.json` (and any `tsconfig.*.json` variant)
- `keystatic.config.ts`
- `.env*` — all environment files (`.env`, `.env.local`, `.env.production`, `.env.example`, etc.)
- `.nvmrc`
- `.tool-versions`
- Any `index.ts` / `index.tsx` barrel file (see §3 — these are auto-generated)
- Everything inside `.claude/` (settings, commands, subagent definitions, hooks)
- `scripts/gen-trail-register.ts` — prebuild hook that writes `src/content/trail-register/commits.json` from `git log`
- `src/content/trail-register/commits.seed.json` — committed fallback data for shallow-clone Vercel builds (the generated `commits.json` is gitignored)

If a subagent needs a change to any of these files, it must surface the requested change to the orchestrator instead of editing directly. The orchestrator applies the change and any dependent updates as a single commit.

## 3. Auto-generated files

The following files are generated by scripts and must not be hand-edited. Changes will be silently overwritten on the next generation pass.

- `src/**/index.ts` and `src/**/index.tsx` — barrel files regenerated by `scripts/gen-barrels.ts`

To change what a barrel exports, edit the source files inside its folder and re-run the generator.

## 4. Parallel-safe directories

Subagents may create, edit, and delete files freely inside these directories, provided they respect the single-owner convention for subfolders described below:

- `src/components/<new-component>/` — one subfolder per component. An agent assigned a component owns its entire subfolder (component file, styles, tests, stories). Do not touch another agent's component folder.
- `src/content/` — MDX/Markdown content and collection entries.
- `tests/` — Vitest unit tests, Playwright specs, visual regression baselines, fixtures.
- `docs/` — design plans, ADRs, internal notes.

Even inside a parallel-safe directory, the rule is: **one agent per subfolder**. Two agents writing into the same `src/components/Hero/` folder concurrently will produce merge conflicts. The orchestrator must route concurrent work to distinct subfolders or serialize the tasks.

Any file or path not listed in §4 and not listed in §2–§3 is orchestrator-owned by default. Ask first.

## 5. Verification gates

No PR reaches Korab until `pnpm verify:all` passes locally. That script chains the four-layer verification pyramid:

1. **Static checks — `pnpm verify`**
   - Biome 2.x: `biome check`
   - TypeScript strict: `tsc --noEmit`

2. **Unit tests — `pnpm test`**
   - Vitest 2.x against `tests/` (non-visual, non-E2E suites)

3. **Visual + E2E — `pnpm test:visual`**
   - Playwright 1.56+ (Agents mode)
   - Screenshots at 375 / 768 / 1280 / 1920 px
   - pixelmatch / odiff-bin as the primary diff check (deterministic)
   - Vision model invoked only on regions where pixel diff > 0.5 %

4. **Performance + accessibility — `pnpm audit`**
   - Lighthouse CI: performance ≥ 0.90, accessibility ≥ 0.95, LCP ≤ 2500 ms, CLS ≤ 0.10
   - axe-core via `@axe-core/playwright`: zero critical or serious violations

All four scripts are live. A subagent whose output cannot be verified end-to-end with `pnpm verify:all` is not shippable, regardless of how correct it looks.

**Post-prompt audit (mandatory):** After completing each playbook prompt, spawn a Sonnet subagent with the prompt's acceptance criteria before moving to the next prompt. The subagent should run `pnpm verify` and `pnpm test` and report a PASS/FAIL table. Do not advance until the audit passes.

**AGENTS.md maintenance:** After completing a prompt, scan this file for stale language — phrases like "not yet", "Prompt N creates", "wired in Prompt N" — and update the relevant section to reflect current reality.

## 6. Escalation policy

Local-model tasks (LM Studio Qwen3.6-35B-A3B, Gemma 4 26B via Ollama) are the default workhorse. Escalation to Anthropic models runs through headless `claude -p` against the Max subscription — no API keys, no separate billing.

**Three-strike rule** (keyed on error fingerprint: error class + file path + line number, stripped of dynamic timestamps and PIDs):

1. **First failure** — retry once with the same prompt, expanded context (add surrounding files, failing test body, recent git diff).
2. **Second failure with the same fingerprint** — retry once more with even more context, or swap to the secondary local model (Gemma if Qwen failed, or vice versa).
3. **Third failure with the same fingerprint** — invoke `bash scripts/escalate.sh "failure description"`.

**Escalation scripts** (all live in `scripts/`):

| Script | Purpose | Model |
|---|---|---|
| `prune-context.sh` | Compresses raw error context to ≤500 tokens | Haiku |
| `escalate.sh` | Prunes then escalates to Sonnet; pass `--model opus` for critical failures | Sonnet / Opus |
| `rate-limit-guard.sh` | Sourceable library — `claude_guarded()` with 3 retries, 1s sleep | — |
| `commit-message.sh` | Generates conventional commit messages from `git diff --cached` | Haiku |

**Always prune before escalating.** `escalate.sh` does this automatically. If calling `claude -p` directly, pipe through `prune-context.sh` first.

Hard rules:

- **Never send raw Playwright traces to Opus or Sonnet.** Prune first.
- **Never send full DOM dumps.** Prune first.
- **Never send complete stack traces.** Prune first.
- The pruner's summary is capped at ~500 tokens. If that is not enough, shrink the question, not the limit.

**Escalation targets:**
- Default: `bash scripts/escalate.sh "description"` → Sonnet
- Critical path (auth, deploy, data corruption): `bash scripts/escalate.sh --model opus "description"`

All escalations count against the Claude Max weekly limit. The pruner is the single biggest lever for staying under that cap — do not skip it.

## 7. Skill routing

This repo operates autonomously. The project-scoped contextual skills at `.claude/skills/` (from `addyosmani/agent-skills`, 21 skills covering debugging, shipping, review, planning, security, performance, UI, testing, docs, migrations, etc.) are the default layer. They fire on conversation context, not on explicit invocation.

Rules:

- **Prefer project-scoped contextual skills for every workflow.** When a context trigger matches one of the 21 skills, invoke it — do not pause to ask.
- **Never proactively invoke or suggest gstack skills** — the global skills registered from `~/.claude/skills/` (e.g. `/investigate`, `/ship`, `/review`, `/qa`, `/design-review`, `/plan-ceo-review`, `/plan-eng-review`, `/plan-design-review`, `/autoplan`, `/office-hours`, `/health`, `/document-release`, `/context-save`, `/context-restore`, `/design-consultation`, `/cso`, `/retro`, `/canary`, etc.). Korab triggers these manually with `/name` when he wants them; treat them as opt-in only.
- **The parent `Korab's Personal Brand/CLAUDE.md` `## Skill routing` block does not apply in this repo.** That block routes errors → `/investigate`, ship → `/ship`, UI audit → `/design-review`, etc. This §7 supersedes it for everything under `Dev Team/`.
- **When a project-scoped skill and a gstack skill both match a context, the project-scoped one wins.** That is intentional. Do not invoke both.
- **No "are you sure?" prompts on routine actions.** Run the contextual skill, commit, push, open the PR. Only pause for destructive operations (§6-adjacent: data loss, shared-state writes, force-push, etc.) — the standard high-blast-radius rules from global CLAUDE.md still apply.
